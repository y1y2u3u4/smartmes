## 二、详细PRD内容大纲

### 1. 产品概述

**产品名称**：SmartMES Lite - 半导体生产工单跟踪系统

**一句话介绍**：
面向半导体制造企业的轻量级工单跟踪与异常管理系统，帮助生产管理人员实时掌握工单进度、快速响应异常停机，提升生产效率和设备利用率。

**目标用户**：
- 生产车间主管/班组长
- 工艺工程师
- 设备维护工程师
- 质量管理人员
- 生产计划调度员

**核心价值主张**：
- ✅ **实时可见**：工单状态、设备状态、异常情况一目了然
- ✅ **快速响应**：异常停机5分钟内完成上报与响应
- ✅ **数据追溯**：批次、设备、工艺参数全程可追溯
- ✅ **简单易用**：传统制造业UI风格，无需培训即可上手
- ✅ **低成本部署**：基于Spring Boot + Vue的轻量级架构

### 2. 行业背景与问题分析

#### 2.1 半导体制造行业特点

| 特点 | 说明 | 系统要求 |
|------|------|----------|
| **高精度** | 工艺参数要求极高（纳米级） | 精确记录每个工艺步骤的参数 |
| **强追溯** | 需要追溯到每片晶圆的完整历史 | 批次号、工单号、设备号全程绑定 |
| **多工序** | 一个产品需要经过数百道工序 | 支持复杂的工序流转和状态管理 |
| **高自动化** | 大量自动化设备 | 需要与设备系统（SECS/GEM）对接 |
| **严格合规** | ISO、IATF等质量体系要求 | 审计日志、电子签名、权限管理 |

#### 2.2 当前痛点

**问题1：工单信息不透明**
- 现状：依赖纸质工单或Excel表格，信息滞后
- 影响：无法实时了解工单进度，影响生产调度

**问题2：异常响应慢**
- 现状：设备停机后，通过电话或微信上报，响应时间长
- 影响：停机时间延长，影响交期和成本

**问题3：数据追溯困难**
- 现状：历史数据分散在各个系统，查询困难
- 影响：质量问题发生时，无法快速定位原因

**问题4：系统复杂度高**
- 现状：大型MES系统功能过于复杂，部署周期长、成本高
- 影响：中小企业无法承担

#### 2.3 竞品分析

基于市场调研，2025年主流MES系统特点：

| 竞品 | 优势 | 劣势 | 我们的差异化 |
|------|------|------|--------------|
| **鼎捷MES** | 功能全面，半导体行业市占率高 | 价格高，部署周期长（6-12个月） | 轻量级，2周内可部署 |
| **宝信软件MES** | 深耕流程工业，设备对接能力强 | 定制化程度高，维护成本高 | 标准化产品，低维护成本 |
| **金蝶云·星空MES** | 移动端支持好（微信小程序） | 通用制造业为主，半导体特性弱 | 专注半导体行业特性 |
| **Critical Manufacturing** | 国际领先，完整可追溯性 | 国外产品，本地化支持弱 | 本地化部署，中文界面 |

### 3. 产品功能需求（MVP）

#### 3.1 工单管理模块（P0）

**功能描述**：管理生产工单的全生命周期，从创建到完工。

**核心功能**：

| 功能项 | 描述 | 优先级 |
|--------|------|--------|
| **工单创建** | 录入工单号、产品型号、批次号、计划数量、交期 | P0 |
| **工单派工** | 分配到具体产线、设备、班组 | P0 |
| **工序流转** | 记录每道工序的开始/完成时间、操作人员 | P0 |
| **进度跟踪** | 实时显示工单完成进度（已完成数量/计划数量） | P0 |
| **工单关闭** | 完工确认、质检通过后关闭工单 | P0 |
| **工单查询** | 按工单号、批次号、日期范围查询 | P0 |

**界面设计要点**：
- 表格列表显示，支持排序、筛选
- 使用颜色标识工单状态（进行中=蓝色、已完成=绿色、异常=红色）
- 操作按钮布局清晰，避免误操作

**数据字段**（核心）：
```
work_order (工单表)
- order_id: 工单号（主键）
- product_code: 产品编码
- batch_no: 批次号
- plan_qty: 计划数量
- actual_qty: 实际完成数量
- start_time: 开始时间
- end_time: 结束时间
- status: 状态（待开始/进行中/已完成/已关闭/异常）
- line_id: 产线ID
- equipment_id: 设备ID
- operator_id: 操作员ID
- created_by: 创建人
- created_at: 创建时间
- updated_at: 更新时间
```

#### 3.2 异常停机上报模块（P0）

**功能描述**：快速上报设备异常停机，记录原因和处理过程。

**核心功能**：

| 功能项 | 描述 | 优先级 |
|--------|------|--------|
| **异常上报** | 选择设备、异常类型、填写描述 | P0 |
| **异常分类** | 设备故障、物料缺料、质量异常、人员操作失误等 | P0 |
| **响应记录** | 记录响应人员、响应时间、处理措施 | P0 |
| **异常关闭** | 确认问题解决，记录停机时长 | P0 |
| **异常统计** | 按设备、类型、时间段统计异常次数和停机时长 | P0 |
| **异常预警** | 同一设备短期内多次异常时发出预警（未来） | P1 |

**界面设计要点**：
- 上报界面简洁，3步完成上报（选设备 → 选类型 → 填描述）
- 支持拍照上传（设备异常现场照片）
- 异常列表按紧急程度排序（红色=紧急、黄色=一般）

**数据字段**（核心）：
```
downtime_report (停机上报表)
- report_id: 上报ID（主键）
- order_id: 关联工单号
- equipment_id: 设备ID
- downtime_type: 停机类型（设备故障/物料缺料/质量异常/人员失误）
- description: 异常描述
- start_time: 停机开始时间
- end_time: 停机结束时间
- duration_minutes: 停机时长（分钟）
- reporter_id: 上报人
- responder_id: 响应人
- solution: 处理措施
- status: 状态（待处理/处理中/已解决）
- attachments: 附件（照片路径）
- created_at: 创建时间
```

**异常类型预设**：
- 设备故障：机械故障、电气故障、软件故障、传感器异常
- 物料问题：缺料、材料不良、上料错误
- 质量异常：产品不良、参数超标
- 人员问题：操作失误、人员不足
- 其他

#### 3.3 数据看板模块（P0）

**功能描述**：实时展示生产关键指标，支持管理决策。

**核心看板**：

**看板1：生产概览**
- 今日工单总数 / 已完成 / 进行中 / 异常
- 今日计划产量 / 实际产量 / 完成率
- 今日设备运行状态（运行/空闲/故障）

**看板2：异常监控**
- 今日异常次数 / 总停机时长
- 异常类型分布（饼图）
- 设备故障TOP5（柱状图）

**看板3：工单进度**
- 按产线显示工单进度（甘特图或进度条）
- 预警：接近交期但进度落后的工单（红色标识）

**界面设计要点**：
- 采用传统ERP风格的蓝白配色
- 数据刷新频率：30秒自动刷新
- 支持大屏显示模式（车间电视墙）

#### 3.4 基础数据管理（P0）

**设备档案管理**：
- 设备编号、设备名称、设备类型、所属产线
- 设备状态（运行/停机/保养）
- 维护记录（最后维护时间、下次计划维护时间）

**产品工艺路线**：
- 产品编码、产品名称
- 工艺步骤列表（工序号、工序名称、标准工时）

**人员管理**：
- 员工工号、姓名、岗位、班组
- 操作权限（普通员工/班组长/工程师/管理员）

### 4. 非功能需求

#### 4.1 性能要求

| 指标 | 要求 | 说明 |
|------|------|------|
| 响应时间 | 页面加载 < 2秒 | 局域网环境 |
| 并发用户 | 支持100人同时在线 | 满足中型工厂需求 |
| 数据刷新 | 看板30秒自动刷新 | 实时性要求 |
| 数据库性能 | 单表100万条记录，查询 < 3秒 | 支持1年历史数据 |

#### 4.2 安全与合规

| 要求 | 实现方式 |
|------|----------|
| **用户认证** | Spring Security + JWT Token |
| **权限控制** | 基于角色的权限管理（RBAC） |
| **操作审计** | 所有数据变更记录操作人、时间、IP |
| **数据备份** | 每日自动备份数据库 |
| **电子签名** | 关键操作（工单关闭、异常确认）需要电子签名 |

#### 4.3 可靠性要求

- **系统可用性**：99.5%（允许每月停机3.6小时）
- **数据完整性**：采用事务机制，确保数据一致性
- **容错能力**：网络中断时，支持离线上报（缓存到本地，恢复后同步）

#### 5.2 核心技术栈

**后端技术栈**（基于Java）：
- **框架**：Spring Boot 3.x + Spring MVC
- **安全**：Spring Security + JWT
- **ORM**：MyBatis Plus（推荐）或 Spring Data JPA
- **数据库**：MySQL 8.0
- **缓存**：Redis（可选，用于看板数据缓存）
- **定时任务**：Spring Scheduler（数据统计、自动备份）

**前端技术栈**（基于Vue）：
- **框架**：Vue 3 + Vue Router + Vuex/Pinia
- **UI组件库**：Element Plus（传统ERP风格）
- **图表**：ECharts（数据看板）
- **HTTP客户端**：Axios
- **构建工具**：Vite

**部署方案**：
- **Web服务器**：Nginx（反向代理 + 静态资源）
- **应用服务器**：Tomcat（内嵌于Spring Boot）
- **数据库**：MySQL 8.0（独立部署）
- **部署环境**：Linux（CentOS 8 / Ubuntu 22.04）
- **容器化**（可选）：Docker + Docker Compose

#### 5.3 数据库设计（核心表）

**工单表（work_order）**：
```sql
CREATE TABLE work_order (
    order_id VARCHAR(50) PRIMARY KEY COMMENT '工单号',
    product_code VARCHAR(50) NOT NULL COMMENT '产品编码',
    batch_no VARCHAR(50) COMMENT '批次号',
    plan_qty INT NOT NULL COMMENT '计划数量',
    actual_qty INT DEFAULT 0 COMMENT '实际完成数量',
    start_time DATETIME COMMENT '开始时间',
    end_time DATETIME COMMENT '结束时间',
    status VARCHAR(20) NOT NULL COMMENT '状态',
    line_id VARCHAR(50) COMMENT '产线ID',
    equipment_id VARCHAR(50) COMMENT '设备ID',
    operator_id VARCHAR(50) COMMENT '操作员ID',
    created_by VARCHAR(50) NOT NULL COMMENT '创建人',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_batch_no (batch_no),
    INDEX idx_status (status),
    INDEX idx_created_at (created_at)
) COMMENT='生产工单表';
```

**停机上报表（downtime_report）**：
```sql
CREATE TABLE downtime_report (
    report_id BIGINT AUTO_INCREMENT PRIMARY KEY,
    order_id VARCHAR(50) COMMENT '关联工单号',
    equipment_id VARCHAR(50) NOT NULL COMMENT '设备ID',
    downtime_type VARCHAR(50) NOT NULL COMMENT '停机类型',
    description TEXT COMMENT '异常描述',
    start_time DATETIME NOT NULL COMMENT '停机开始时间',
    end_time DATETIME COMMENT '停机结束时间',
    duration_minutes INT COMMENT '停机时长（分钟）',
    reporter_id VARCHAR(50) NOT NULL COMMENT '上报人',
    responder_id VARCHAR(50) COMMENT '响应人',
    solution TEXT COMMENT '处理措施',
    status VARCHAR(20) NOT NULL COMMENT '状态',
    attachments VARCHAR(500) COMMENT '附件路径',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_equipment (equipment_id),
    INDEX idx_status (status),
    INDEX idx_start_time (start_time),
    FOREIGN KEY (order_id) REFERENCES work_order(order_id)
) COMMENT='停机异常上报表';
```

### 6. UI/UX设计原则（传统制造业风格）

#### 6.1 设计风格定位

**关键词**：简洁、实用、稳定、高效

**参考对象**：传统ERP系统（SAP、金蝶、用友）的界面风格

#### 6.2 设计规范

**色彩方案**：
- 主色调：深蓝色（#1E3A8A）- 专业、稳定
- 辅助色：灰色（#6B7280）- 中性、低干扰
- 成功色：绿色（#10B981）- 正常、完成
- 警告色：黄色（#F59E0B）- 注意、一般异常
- 危险色：红色（#EF4444）- 紧急、严重异常

**布局原则**：
- 顶部导航栏：Logo + 主菜单 + 用户信息
- 左侧菜单栏（可折叠）：功能模块树状结构
- 主内容区：面包屑导航 + 操作按钮区 + 数据表格/表单
- 底部：版权信息 + 在线人数

**表格设计**：
- 使用斑马纹（隔行变色）提升可读性
- 表头固定（数据滚动时表头不动）
- 操作列固定在最右侧
- 支持列排序、筛选、导出Excel

**表单设计**：
- 字段标签左对齐，输入框右侧
- 必填字段用红色*标识
- 提供字段说明（灰色小字）
- 提交按钮使用主色调，取消按钮使用灰色

**响应式设计**（优先级低）：
- MVP阶段主要面向PC端（1920x1080分辨率）
- 未来支持平板端（iPad）用于车间现场操作

### 7. 实施路线图

#### Phase 1: MVP阶段（6-8周）

**Week 1-2: 需求确认与设计**

## 四、技术难点与解决方案

### 难点1：实时数据刷新

**问题**：看板数据需要30秒自动刷新，如何避免频繁查询数据库影响性能？

**解决方案**：
- 使用Redis缓存看板数据，TTL设为30秒
- 后台定时任务（Spring Scheduler）每30秒更新缓存
- 前端使用轮询（setInterval）或WebSocket获取最新数据

### 难点2：并发写入冲突

**问题**：多个用户同时更新同一工单的实际数量，如何避免数据冲突？

**解决方案**：
- 使用乐观锁（version字段）
- 数据库层面使用事务隔离级别（READ COMMITTED）
- 前端提示用户刷新后重试

### 难点3：大数据量查询性能

**问题**：工单表1年后数据量达到100万条，查询慢怎么办？

**解决方案**：
- 数据库索引优化（order_id、batch_no、created_at、status）
- 分表策略（按月份分表，如work_order_202501）
- 历史数据归档（1年前的数据迁移到历史表）

### 难点4：离线上报支持

**问题**：车间网络不稳定，如何确保异常上报不丢失？

**解决方案**：
- 前端使用IndexedDB缓存离线数据
- 网络恢复后自动同步到服务器
- 使用时间戳去重，避免重复提交
